## 1. Summary of Changes

v1.1 introduces **5 new columns across 3 tables** and **no table drops or column renames**.

All changes are backward-compatible:
- New columns are `NULLABLE` or carry safe `DEFAULT` values.
- No existing column types are modified.
- No existing indexes are removed.
- v1.0 application code can run against the v1.1 schema without errors (new columns are ignored).

| Table | Change | Type | Default | Nullable |
|-------|--------|------|---------|----------|
| `messages` | Add `emotion_tag` | `VARCHAR(20)` | — | YES |
| `messages` | Add `importance_score` | `FLOAT` | `0.5` | NO |
| `chat_sessions` | Add `session_summary` | `TEXT` | — | YES |
| `chat_sessions` | Add `summary_updated_at` | `TIMESTAMPTZ` | — | YES |
| `chat_sessions` | Add `title_generated` | `BOOLEAN` | `false` | NO |
| `users` | Add `memory_ttl_preference` | `VARCHAR(20)` | `'24h'` | YES |

> **Note:** v1.1 is split into **two migrations** (002 and 003) for independent rollback capability. Migration 002 covers data model changes. Migration 003 covers future index additions if query patterns change.

## 2. Schema Changes by Table

### 2.1 `messages` Table

**Current schema (v1.0):**
```sql
CREATE TABLE messages (
    id               UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id       UUID         NOT NULL REFERENCES chat_sessions(id) ON DELETE CASCADE,
    role             VARCHAR(20)  NOT NULL,
    content          TEXT         NOT NULL,
    relevance_score  FLOAT,
    is_crisis        BOOLEAN      NOT NULL DEFAULT false,
    created_at       TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

**v1.1 additions:**

| Column | Type | Constraint | Rationale |
|--------|------|-----------|-----------|
| `emotion_tag` | `VARCHAR(20)` | `NULL` | Ekman 7-class emotion detected at write time. `NULL` when tagger is unavailable or confidence below threshold. Allowed values: `joy`, `sadness`, `fear`, `anger`, `disgust`, `surprise`, `neutral`. |
| `importance_score` | `FLOAT` | `NOT NULL DEFAULT 0.5` | Heuristic importance score (0.0–1.0). Default 0.5 for historical rows (neutral assumption). Used for memory eviction priority and analytics. |

**Target schema (v1.1):**
```sql
CREATE TABLE messages (
    id               UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id       UUID         NOT NULL REFERENCES chat_sessions(id) ON DELETE CASCADE,
    role             VARCHAR(20)  NOT NULL,
    content          TEXT         NOT NULL,
    relevance_score  FLOAT,
    is_crisis        BOOLEAN      NOT NULL DEFAULT false,
    emotion_tag      VARCHAR(20),                         -- NEW
    importance_score FLOAT        NOT NULL DEFAULT 0.5,  -- NEW
    created_at       TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

**Check constraint (new):**
```sql
ALTER TABLE messages
  ADD CONSTRAINT ck_messages_emotion_tag
  CHECK (emotion_tag IN ('joy','sadness','fear','anger','disgust','surprise','neutral') OR emotion_tag IS NULL);
```

**Check constraint (new):**
```sql
ALTER TABLE messages
  ADD CONSTRAINT ck_messages_importance_score
  CHECK (importance_score >= 0.0 AND importance_score <= 1.0);
```

### 2.2 `chat_sessions` Table

**Current schema (v1.0):**
```sql
CREATE TABLE chat_sessions (
    id             UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id        UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title          VARCHAR(255) NOT NULL DEFAULT 'New Conversation',
    message_count  INTEGER      NOT NULL DEFAULT 0,
    crisis_flagged BOOLEAN      NOT NULL DEFAULT false,
    started_at     TIMESTAMPTZ  NOT NULL DEFAULT now(),
    ended_at       TIMESTAMPTZ
);
```

**v1.1 additions:**

| Column | Type | Constraint | Rationale |
|--------|------|-----------|-----------|
| `title_generated` | `BOOLEAN` | `NOT NULL DEFAULT false` | Tracks whether the title was AI-generated (true) or fallback-truncated (false). Used to avoid re-generating titles and for analytics. |
| `session_summary` | `TEXT` | `NULL` | Rolling summary generated by Gemini every 20 messages. Injected into LLM prompt for long-session context continuity. `NULL` until first trigger fires. |
| `summary_updated_at` | `TIMESTAMPTZ` | `NULL` | Timestamp of last summary generation. `NULL` until first summary exists. Used to track freshness. |

**Target schema (v1.1):**
```sql
CREATE TABLE chat_sessions (
    id                  UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title               VARCHAR(255) NOT NULL DEFAULT 'New Conversation',
    title_generated     BOOLEAN      NOT NULL DEFAULT false,           -- NEW
    message_count       INTEGER      NOT NULL DEFAULT 0,
    crisis_flagged      BOOLEAN      NOT NULL DEFAULT false,
    session_summary     TEXT,                                          -- NEW
    summary_updated_at  TIMESTAMPTZ,                                   -- NEW
    started_at          TIMESTAMPTZ  NOT NULL DEFAULT now(),
    ended_at            TIMESTAMPTZ
);
```

### 2.3 `users` Table

**Current schema (v1.0):**
```sql
CREATE TABLE users (
    id               UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    email            VARCHAR(255) NOT NULL UNIQUE,
    hashed_password  VARCHAR(255) NOT NULL,
    full_name        VARCHAR(255),
    is_verified      BOOLEAN      NOT NULL DEFAULT false,
    is_active        BOOLEAN      NOT NULL DEFAULT true,
    emergency_email  VARCHAR(255),
    created_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at       TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

**v1.1 addition:**

| Column | Type | Constraint | Rationale |
|--------|------|-----------|-----------|
| `memory_ttl_preference` | `VARCHAR(20)` | `NULL DEFAULT '24h'` | User-configurable Redis TTL preference. Accepted values: `24h`, `72h`, `7d`, `session`. Application-layer validation enforces the allowlist. `NULL` treated as `24h` by application code. |

**Check constraint (new):**
```sql
ALTER TABLE users
  ADD CONSTRAINT ck_users_memory_ttl_preference
  CHECK (memory_ttl_preference IN ('24h', '72h', '7d', 'session') OR memory_ttl_preference IS NULL);
```

**Target schema (v1.1):**
```sql
CREATE TABLE users (
    id                      UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    email                   VARCHAR(255) NOT NULL UNIQUE,
    hashed_password         VARCHAR(255) NOT NULL,
    full_name               VARCHAR(255),
    is_verified             BOOLEAN      NOT NULL DEFAULT false,
    is_active               BOOLEAN      NOT NULL DEFAULT true,
    emergency_email         VARCHAR(255),
    memory_ttl_preference   VARCHAR(20)  DEFAULT '24h',              -- NEW
    created_at              TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at              TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

def upgrade() -> None:
    # Index for filtering messages by emotion_tag (analytics queries)
    op.create_index(
        "idx_messages_emotion_tag",

## 5. Rollback Strategy

### Primary Rollback: Alembic Downgrade

The migration downgrade path is fully specified and tested. To revert v1.1 schema changes:

```bash
# Step 1: Stop the application
# Step 2: Run Alembic downgrade
cd backend
alembic downgrade 001

# Step 3: Verify schema
psql $DATABASE_URL -c "\d messages"
psql $DATABASE_URL -c "\d chat_sessions"
psql $DATABASE_URL -c "\d users"

# Step 4: Redeploy v1.0 application code
```

**Data loss on downgrade:**
- `messages.emotion_tag` — column and data dropped. No recovery path.
- `messages.importance_score` — column and data dropped. No recovery path.
- `chat_sessions.session_summary` — column and data dropped. Summaries lost. Sessions themselves are unaffected.
- `chat_sessions.title_generated` — column dropped. Title text is preserved in `title` column.
- `chat_sessions.summary_updated_at` — column and data dropped.
- `users.memory_ttl_preference` — column and data dropped. Users revert to system default (24h).

> **Important:** The `title` column itself is not dropped. AI-generated titles remain in the `title` column even after downgrade. Only the `title_generated` flag is lost.

### Pre-Migration Backup (Required)

Before running migration 002 in production, take a PostgreSQL dump:

```bash
pg_dump $DATABASE_URL \
  --format=custom \
  --no-owner \
  --file=mindbridge_pre_v1.1_$(date +%Y%m%d_%H%M%S).dump
```

Keep the dump for at least 30 days post-migration in case rollback is needed.

### Rolling Back Redis Schema

Redis entries written by v1.1 code (with `emotion_tag` and `importance_score` fields) are backward-compatible with v1.0 code. The v1.0 `MemoryService` reads Redis entries as raw JSON and accesses only `role` and `content` keys — extra keys are ignored.

No Redis data migration or rollback is required.

### Partial Feature Rollback

If only a specific feature needs to be disabled (without reverting the schema):

| Feature | Rollback action |
|---------|----------------|
| FR-01 Emotion Tagging | Set `EMOTION_CONFIDENCE_THRESHOLD = 2.0` in `.env` — tagger always returns `null` |
| FR-04 Rolling Summaries | Set `SUMMARY_TRIGGER_EVERY = 0` — trigger never fires |
| FR-06 AI Titles | Set env var `DISABLE_AI_TITLES=true` and add check in `title_service.py` |
| FR-05 Existential Guard | Set `EXISTENTIAL_MARGIN = 0.0` — guard never activates |
| FR-07 Warm Load | Remove `warm_load()` calls from chat.py/chat_ws.py — schema unchanged |

## 6. Index Strategy

### Existing Indexes (unchanged)
```
idx_sessions_user          → chat_sessions(user_id)
idx_messages_session       → messages(session_id)
ix_users_email             → users(email) UNIQUE
idx_mood_logs_user         → mood_logs(user_id)
```

### New Indexes (Migration 003, deferred)

| Index | Table | Column | Type | Rationale |
|-------|-------|--------|------|-----------|
| `idx_messages_emotion_tag` | `messages` | `emotion_tag` | Partial B-tree (IS NOT NULL) | Future analytics: emotion trend queries by session |
| `idx_sessions_summary` | `chat_sessions` | `summary_updated_at` | Partial B-tree (IS NOT NULL) | Admin review queries: find sessions with summaries |

### Query Performance Assessment

The core application queries are unchanged in v1.1:
- `SELECT * FROM messages WHERE session_id = $1 ORDER BY created_at LIMIT 20` — uses existing `idx_messages_session`.
- `SELECT * FROM chat_sessions WHERE user_id = $1 ORDER BY started_at DESC LIMIT 50` — uses existing `idx_sessions_user`.

The new `importance_score` and `emotion_tag` columns are read as part of full row fetches — no new sequential scans are introduced by v1.1 application code.

## 7. Data Backfill

### `messages.importance_score`
The `server_default = 0.5` ensures all historical rows receive a valid importance score at DDL time via PostgreSQL's default mechanism. **No explicit UPDATE backfill is needed.**

### `messages.emotion_tag`
Default is `NULL` for all historical rows. Historical messages are not analyzed retroactively (no backfill). This is intentional — the emotion tagger only runs on new messages.

### `chat_sessions.title_generated`
The `server_default = false` ensures all existing sessions are marked as "not AI-generated". This is correct — no existing titles were AI-generated.

### `chat_sessions.session_summary`
Default is `NULL`. Existing sessions do not have summaries. They will accumulate summaries naturally as users continue conversations. No retroactive summarization.

### `users.memory_ttl_preference`
The `server_default = '24h'` ensures all existing users get the system default TTL. No explicit backfill needed.

## 8. Impact Assessment

### Table Growth
- `messages`: +2 columns. Average row size increases by approximately 25 bytes (`emotion_tag` avg 10 bytes, `importance_score` 8 bytes). For a database with 1M messages, this is ~25 MB additional storage — negligible.
- `chat_sessions`: +3 columns. `session_summary` (TEXT) is the significant one — average summary ~400 characters = ~400 bytes per session. For 100K sessions: ~40 MB additional storage.
- `users`: +1 column, 20 bytes max. Negligible.

### Migration Execution Time
Migration 002 runs `ALTER TABLE ... ADD COLUMN` on potentially large tables. PostgreSQL 12+ handles `ADD COLUMN ... DEFAULT` (without volatile expressions) as a metadata-only operation — it does **not** rewrite the table. Expected migration time: < 5 seconds regardless of table size.

### Application Downtime
Zero downtime migration is possible:
1. Apply migration 002 while v1.0 app is running (new columns are nullable/defaulted; v1.0 ignores them).
2. Deploy v1.1 application code.
3. v1.1 code begins writing to new columns immediately.

The only risk is a brief window where v1.1 code runs but migration has not yet been applied. To prevent this, always apply migrations before deploying new code.

### SQLAlchemy ORM Impact
The v1.1 SQLAlchemy models are the source of truth for column definitions. Alembic's `autogenerate` can be used to detect drift between models and database:

```bash
cd backend
alembic check  # raises error if models differ from db schema
```

Run `alembic check` as part of the CI pipeline to catch schema drift early.
